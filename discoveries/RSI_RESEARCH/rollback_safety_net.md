# 整合16：回滚安全网 — ACE+Galileo 模式落地

**日期**: 2026-05-06  
**触发**: 6步探索循环第3轮  
**关联**: 整合15（自修改提案系统）→ 安全层补充  

---

## 问题

整合15创建了自修改提案系统，但退化检测只有一个粗粒度条件："连续3次循环失败"。这无法捕获：
- 渐进性能劣化（循环越来越慢，但没完全失败）
- 单次灾难性修改（一次改坏，立即影响后续所有循环）
- 静默退化（修改后循环正常跑，但产出质量下降）

## 模式参考

### ACE playbook
```
检测（±0.4%漂移）→ 提炼（标记为具体失败模式）→ 持久化（计数器累加）→ 重用（后续任务免疫）
回滚：性能回归自动触发，<5分钟完成
性能预算：≤10ms P50 回放检索，≤+15% 端到端开销
```

### Galileo
```
eval-driven guardrail：离线评估指标 → 在线 guardrail 策略
<200ms 实时拦截
信号检测：自动发现未知故障模式
```

## 落地设计

### 三条检测线

| 检测线 | 条件 | 触发动作 | 参考 |
|:---:|---|---|:---:|
| **红线** | 连续 3 次循环失败（无产出） | 立即回滚到上一个稳定快照，2周期冷却 | 已有 |
| **黄线** | 连续 5 次循环耗时递增（每次 +10%+） | 标记退化警报，暂停新提案提交，进入观察模式 | ACE 漂移检测 |
| **蓝线** | 单次修改后首周期耗时 > 基线的 150% | 立即回滚该次修改（不等待连续失败），1周期冷却 | Galileo 实时拦截 |

### 基线定义
- **绝对基线**：修改前的最近 5 次正常循环的平均耗时
- **相对基线**：修改前循环的平均产出质量（可观测痕迹数）

### 实现

1. `staging/heartbeat_proposals.md` 追加 `## 退化检测状态` 章节
   - 记录每次循环耗时
   - 维护 5 次滑动窗口
   - 三线状态追踪

2. cron [1]步骤（阅读清单）增加退化检查
   - 读取上次循环耗时
   - 与基线对比
   - 触发对应检测线

3. 回滚动作
   - 恢复到上一个稳定快照的 HEARTBEAT.md
   - 清空提案队列中来源为回滚修改的提案
   - 记录回滚事件

## 与现有系统的集成

```
cron [1] 阅读清单
    ↓
退化检测（三线判断）
    ↓
├─ 红线触发 → 自动回滚 → 冷却 → 跳过[2]-[6]
├─ 黄线触发 → 暂停提案 → 观察模式 → [2]-[5]跳过[6]（不产生新提案）
├─ 蓝线触发 → 回滚上一提案 → 冷却 → 跳过[2]-[6]
└─ 正常      → 继续[2]-[6]
```
