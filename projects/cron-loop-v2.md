## 循环结构（6步闭环）

```
cron触发
    ↓
[1] 阅读清单 ← 读 MEMORY.md / HEARTBEAT.md / 相关SKILL → 知道"现在该探索什么"
    ↓
[2] 探索 ← 搜索/阅读外部信息 → 获取原始素材
    ↓
[3] 总结 ← 提炼洞察（不是写报告，是问自己：这和清单有什么关系？）
    ↓
[4] 行动 ← 基于洞察做可观测的事（修改文件、发消息、创建产品）
    ↓
[5] 报告 ← 行动的结果，不是探索的总结
    ↓
[6] 更新清单 ← 把教训/发现/新指针固化到系统（MEMORY.md/HEARTBEAT.md/SKILL）
    ↓
回到 [1] 等待下次触发
```

**关键修正**：
- ❌ 旧模式: 探索 → 写报告 → 结束（循环断裂）
- ✅ 新模式: 探索 → 行动 → 更新清单 → 结束（循环闭环）

---

## 每步的产出和流向

| 步骤 | 产出 | 流向 |
|------|------|------|
| 阅读清单 | 知道"此刻缺口在哪" | 指导探索方向 |
| 探索 | 原始素材/搜索结果 | 进入总结 |
| 总结 | 一句话洞察 + 行动方向 | 驱动行动 |
| 行动 | 可观测痕迹（文件/消息/commit） | 生成报告 |
| 报告 | 行动结果的简述 | 触发清单更新 |
| 更新清单 | 修改后的系统文件 | 下次循环的输入 |

---

## 为什么旧模式断裂

旧cron指令：
```
步骤2: 搜索 → 步骤3: 写入cron_discoveries.md → 步骤5: 输出简报
```

问题：
1. **没有[1]阅读清单** → 探索是随机的，不是基于当前缺口
2. **没有[4]行动** → 探索直接跳到报告
3. **没有[6]更新清单** → 发现没有固化，下次从零开始
4. **报告是探索的总结** → 不是行动的结果

## cron任务重构（溯-自主探索 v2）

### 触发器设置
- 频率: 每20分钟（保持）
- 轻量上下文: true（减少token消耗）
- thinking: low（减少推理开销）

### 指令结构

```
你是一个循环执行器。每次唤醒，执行完整的6步闭环。

## [1] 阅读清单（30秒）
读取 ~/.openclaw/workspace/MEMORY.md —— 找到"当前缺口"指针
读取 ~/.openclaw/workspace/HEARTBEAT.md —— 找到"此刻该做什么"的线索
如果清单指示暂停/等待，直接输出"[状态] 清单指示等待，跳过本次循环"

## [2] 探索（15-30秒）
基于清单缺口，选择搜索方向：
- 如果缺口是"行业洞察" → 搜索家居建材/AI转型/市场动态
- 如果缺口是"技术前沿" → 搜索AI记忆架构/递归改进/智能体
- 如果缺口是"收入模式" → 搜索被动收入/自由职业/数字产品
使用 kimi_search 搜索2-3条结果。

## [3] 总结（15秒）
问自己：这个发现，如何填补清单中的缺口？
产出：一句话洞察 + 明确的行动方向（修改哪个文件/发什么消息/创建什么）

## [4] 行动（30-60秒）
必须执行可观测痕迹：
- 修改 SKILL.md / HEARTBEAT.md / MEMORY.md
- 创建可交付物到 memory/topics/exploration/
- 发飞书消息（如果有明确接收者）
- 在GitHub上comment/issue
禁止：仅把结果写入临时文件就结束

## [5] 报告（15秒）
格式: "[行动] <做了什么> → 结果: <可观测痕迹> → 清单已更新: <文件>"
字数: ≤60字

## [6] 更新清单（15秒）
- 如果行动成功 → 更新 MEMORY.md 指针（添加新文件路径/修改状态）
- 如果行动失败 → 更新 HEARTBEAT.md 的"迭代记录"（记录教训）
- 如果发现新缺口 → 添加新指针到 MEMORY.md

## 规则
- 完整6步必须执行，不可跳过
- [4]行动失败时，[6]记录失败原因，不要假装成功
- 如果[1]发现清单无缺口，跳过[2]-[5]，仅执行[6]记录"本次跳过"
- 超时120秒
```

## 与action-vs-report skill的关系

- **skill** 是过滤器：判断"这个产出是否算行动"
- **cron循环** 是管道：确保"行动后必须更新清单"

两者配合：
- cron执行到[4]时，用skill检验"这是否算有效行动"
- 如果不算 → 回退到[3]重新选择行动方向
- 如果算 → 继续到[6]更新清单
